## 一、带有外部反馈的反思

### 1. 核心流程步骤

1. **任务发起：** 用户要求为任务 X 编写代码。
2. **撰写初稿 (Write first draft)：** 大模型（LLM）写出第一版代码 **code V1**。
3. **外部执行 (Execute code) —— 关键点：**系统不再是让 AI 盯着代码看，而是将 code V1 发送到一个真实的**代码执行环境**（如 Python 沙箱）中去运行。图中绿色方框代表了这个“实际动手”的过程。
4. **捕获反馈 (Code output, errors)：**执行环境返回真实的结果。图中给出了一个典型的报错案例：SyntaxError: unterminated string literal（语法错误：字符串字面量未结束）。
5. **针对性反思 (Reflect and improve)：**LLM 接收到这个**具体的报错信息**。此时 AI 不再是盲目猜测，而是根据具体的错误提示进行精准“反思”：“哦，我第一行少写了一个引号，导致了语法错误。”
6. **最终产出：** AI 编写出修正后的第二版代码 **do_task_v2**。

### 2. 为什么这种模式比“自我反思”更强大？

- **从“主观猜测”到“客观事实”：** 在纯文字反思中，AI 可能会漏掉细微的语法错误。而在外部反馈模式下，**报错是铁证如山的历史事实**，AI 必须面对并修复它。
- **模拟真实人类开发者的行为：** 优秀的程序员不是一次性写对代码，而是通过“写代码 -> 运行报错 -> 调试改 Bug”的循环来完成工作的。这个模式赋予了 AI 同样的**调试（Debugging）**能力。
- **极高的准确率：** 通过这种“运行-反馈-修改”的闭环，AI 最终产出的代码可运行率接近 100%。

### 总结：这页课件的实操意义

这页课件实际上向我们展示了 **“代码解释器”** 这种功能背后的逻辑架构：

1. **闭环设计：** 成功的智能体应用必须允许 AI **“看到结果”**。
2. **容错机制：** 允许 AI 在第一步犯错，只要你给它配备了能检测错误的工具。
3. **工具驱动认知：** 外部工具（执行环境）提供的报错信息，成为了提升大模型认知和输出质量的“肥料”。

## 二、直接生成模式

**这是大语言模型最基础、最传统的使用方式，通常被称为**“零样本 (Zero-shot)”**或**“一锤子买卖”模式。

### 1. 核心特征：线性与单向

- **工作流：** 提示词 (Prompt) `→→` 大模型 (LLM) `→→` 最终回复 (Response)。
- **特点：**没有中间步骤： 模型接收到指令后，立即尝试完成整个任务。**没有纠错机会：** 模型不能停下来思考，不能查资料，也不能在发现写错后返回去修改。**即兴发挥：** 这就像让一个学生在没有任何草稿、不准涂改的情况下，直接在考卷上写出一篇完美的论文。

### 2. 课件中的两个案例

- **文字创作类：** 用户要求写一篇关于黑洞的文章，模型直接吐出一段完整文本。
- **代码生成类：** 用户要求写一个 Python 复利计算函数，模型直接写出代码块。

### 3. 深度对比

这页课件的潜台词是：**虽然“直接生成”很快很省钱，但在处理复杂任务时，它的局限性非常明显。**

我们可以对比一下：

| 维度       | 直接生成 (本页)                | 智能体工作流 (前面讲的)                        |
| ---------- | ------------------------------ | ---------------------------------------------- |
| **过程**   | 像“即兴演讲”，说错就错了。     | 像“写论文”，有大纲、有调研、有二稿。           |
| **质量**   | 依赖模型原生智力，易产生幻觉。 | 通过**反思和迭代**，质量大幅提升。             |
| **能力**   | 只能利用模型训练时的记忆。     | 能通过**工具调用**（搜索、代码运行）突破限制。 |
| **适用性** | 适合简单、确定的短任务。       | 适合复杂、需要逻辑推理的长任务。               |

## 三、零样本、单样本和少样本提示词

### 1. Zero-shot (零样本提示)：不给例子

- **做法：** 直接下达指令，告诉 AI 任务是什么，但不提供任何参考示例。
- **课件内容：** “转换为 MM/DD/YYYY 格式。输入：{日期}”。
- **特点：** 最简单、最快捷。
- **适用场景：** 任务非常直观，或者使用的是非常强大的模型（如 GPT-4），它已经在大规模预训练中学会了这个任务。
- **风险：** 模型可能会误解你的格式要求（比如它用了连字符 - 而不是斜杠 /）。

### 2. One-shot (单样本提示)：给一个例子

- **做法：** 在指令之后，给 AI 提供**一个**完整的“输入-输出”示例。
- **课件内容：** 增加了一个例子——“输入：Jan 1st, 2025. 输出：01/01/2025”。
- **特点：** 极大地提高了 AI 对输出**格式和风格**的理解。
- **适用场景：** 当你需要模型严格遵循某种特定的排版或语气时。

### 3. Few-shot (少样本提示)：给多个例子

- **做法：** 提供**两个或更多**示例。
- **课件内容：** 提供了两个例子，分别展示了月份在前和日期在前的情况，帮助 AI 理解各种可能的输入。
- **特点：** **最稳健的方法**。通过多个例子，模型能够识别出更复杂的模式和潜在的边界情况。
- **适用场景：**处理复杂的逻辑任务。需要模型学习非常冷门的知识或特定的公司内部格式。模型较小或较弱，需要额外的引导。

### 核心总结与实操建议

这页课件传达了 AI 使用的一个核心真理：**“给例子比给指令更有效。”**

1. **引导模型学习模式：** 如果你发现 AI 的回答总是差那么一点意思，最快的方法就是给它两个例子（Few-shot）。
2. **节省空间 vs 提高准确性：**例子越多，消耗的 **Token（字符）** 越多，成本越高；但例子越多，**准确性**通常越高，且能有效减少“幻觉”。
3. **开发中的应用：** 在构建智能体（Agent）或 RAG 系统时，开发者通常会在后台提示词中放入几个 Few-shot 示例，以确保系统输出的稳定性。

## 四、反思已被测试证明有效

### 1. 核心结论

> **“反思在各种任务上的表现始终优于直接生成。”**

- **解读：** 这是一个普适性的规律。无论是在写代码、做数学题还是写对话，只要给 AI 一个“复盘和改稿”的机会，它的表现就会产生质的飞跃。

### 2. 图表深度解析

这个柱状图对比了三个模型在不同任务下的表现，每个模型都有两种模式：

- **浅色柱子：** **Base（基础模式）**，即直接生成，不准修改。
- **深色柱子：** **+Self-Refine（反思模式）**，即 AI 写完后进行自我审查和迭代。

#### **关键观察点：**

1. **全面碾压：** 在几乎所有测试任务中，深色柱子都明显高于浅色柱子。
2. **巨大的提升幅度：** 柱子顶部的数字（如 **+49.2**、**+31.8**）代表了百分点的增长。在**对话回复（Dialogue response）**中，GPT-4 通过反思提升了惊人的 **49.2%**。在**受限生成（Constrained generation）**中，ChatGPT 提升了 **23.0%**。
3. **任务覆盖广：**代码类： 代码优化（Code optimization）、代码可读性（Code readability）。**逻辑类：** 数学推理（Math reasoning）。**创意与文本类：** 情感翻转（Sentiment reversal）、首字母缩写生成（Acronym generation）。

## 五、反思表现更好的具体任务场景

### 1. 技术类：生成 HTML 表格

- **常见问题：** **Missing ‘>’ (缺失结束符号)**。**深度解读：** 在生成长代码或复杂标签时，模型可能会因为 Token 预测的微小偏差，导致少写一个括号或尖括号，这会让整段 HTML 失效。
- **反思指令：** **“验证这段 html 代码 (Validate the html code)”**。**作用：** 强迫模型站在“编译器”的角度重新扫描一遍代码。通常在第二遍阅读时，模型能精准发现这种语法级的小失误。

### 2. 逻辑/流程类：如何泡出一杯完美的茶

- **常见问题：** **Missing steps (遗漏步骤)**。**深度解读：** 模型在快速回答时，可能会忽略掉一些对人类来说显而易见但对流程完整性至关重要的细节（比如“等水开”或“温杯”）。
- **反思指令：** **“检查指令的连贯性和完整性 (Check instructions for coherence and completeness)”**。**作用：** 这是一个质量检查。模型会对比它的回答是否符合一个逻辑严密的 SOP（标准作业程序），从而补上那些漏掉的环节。

### 3. 创意/评估类：生成域名

- **常见问题：** **Name has unintended meaning, or is hard to pronounce (名称有歧义，或者难读)**。**深度解读：** 这是一个极其主观且容易“翻车”的领域。模型可能拼凑出一个看起来很好的英文缩写，但这个词在另一种文化里可能是脏话，或者读起来极其绕口。
- **反思指令：** **“这个域名有负面暗示吗？是否朗朗上口？ (Does domain name have any negative connotations? Is the domain name hard to pronounce?)”**。**作用：** 这展示了反思模式最强大的地方——**多维度评估**。通过给模型明确的“审核清单”，它能识破自己刚才生成的那些“看起来不错但实际很烂”的创意。

### 核心总结与实操启示

这页课件教给开发者如何写出好的**反思提示词 (Reflection Prompts)**：

1. **针对性提问：** 针对不同的任务，反思指令要具体。写代码就问“语法对吗”，写创意就问“有歧义吗”。
2. **强制性检查：** 反思模式的本质是给 AI 下达一个**“挑刺”**的任务。
3. **从“能跑”到“好用”：**HTML 的例子是确保**可用性**；泡茶的例子是确保**严谨性**；域名的例子是确保**品味与文化正确性**。

## 六、反思提示词的具体实操建议

### . 案例一：头脑风暴域名

这是一个创意筛选场景。好的反思提示词不是简单说“把名字改好”，而是给出具体的审核维度：

- **明确动作：** “Review the domain names you suggested.”（审查你刚才建议的域名。）
- **具体指标 1（易用性）：** “Check if each name is easy to pronounce...”（检查每个名字是否容易发音，从而便于口碑传播。）
- **具体指标 2（全球化风险）：** “Consider whether each name might mean something negative in other languages.”（考虑每个名字在其他语言中是否可能有负面含义。）
- **结果导向：** “Then output a shortlist...”（然后只输出符合这些标准的精选名单。）

### 2. 案例二：改进电子邮件

这是一个沟通优化场景，侧重于语气和事实的准确性：

- **明确动作：** “Review the email first draft.”（审查邮件初稿。）
- **具体指标 1（专业性）：** “Check that the tone is professional...”（检查语气是否专业，寻找可能被视为无礼或不敏感的词组。）
- **具体指标 2（事实核查）：** “Verify all facts, dates, and promises are accurate.”（验证所有事实、日期和承诺是否准确。）
- **迭代指令：** “Then write the next draft...”（然后编写邮件的下一版草稿。）

### 核心总结：编写反思提示词的两大原则

课件底部的两个点是整页的精髓，也是开发者需要背下来的**“提示词工程准则”**：

1. **Clearly indicate the reflection action（明确指示反思动作）：**不要含糊。使用像 **“Review（审查）”**、**“Check（检查）”**、**“Verify（验证）”** 或 **“Critique（批判）”** 这样具有明确指向性的动词。
2. **Specify criteria to check（指定检查标准）：**AI 需要一张**“核查清单（Checklist）”**。如果你不告诉它检查什么（比如：读音、文化避讳、事实准确性、语气），它可能只会泛泛而谈，无法产出实质性的改进。

## 七、使用不同的模型（或实例）进行交叉审查

### 1. 第一个角色：执行者

- **职责：** 纯粹的任务执行。
- **具体任务：** 根据用户的原始需求，编写 Python 代码来生成可视化图表（即 RAG 或数据分析的初稿）。
- **特点：** 这个模型专注于将自然语言转化为可运行的代码。

### 2. 第二个角色：评审专家 —— 核心创新点

这里不再让 LLM 1 自言自语，而是引入了 LLM 2，并给它设定了极其专业的**角色扮演（Persona）**：

- **设定身份：** “你是一位**资深数据分析师**，负责对可视化图表提供建设性反馈。”
- **全方位输入：** 注意，LLM 2 不仅仅看代码，它接收的上下文非常丰富：{V1 code}：第一版原始代码。{plot.png}：**关键点！** 这是一个多模态反馈。模型需要具备视觉能力，“亲眼看到”图表长什么样（比如：标签是否重叠、颜色是否容易分辨）。{conversation history}：了解用户的初衷。

### 3. LLM 2 的“两步走”工作法

课件展示了给 LLM 2 设计的精妙指令（Prompting）：

- **第一步：挑刺 (Critique)**要求从**可读性、清晰度和完整性**三个专业维度对图表进行“全身体检”。这种“先思考再行动”的模式有助于模型发现隐性错误。
- **第二步：改稿 (Write new code)**在完成了第一步的逻辑批判后，再让它写出改进后的代码。

### 为什么这种“双模型反思”模式更强大？

1. **克服思维定势：** 人类自己检查自己的作文很难发现错别字，AI 也是。LLM 2 作为一个“局外人”，更容易发现 LLM 1 的逻辑漏洞或审美缺陷。
2. **角色力量：** 赋予“资深数据分析师”的身份后，大模型会调动其训练数据中关于“高质量专业图表”的知识，从而给出更严苛、更具水准的建议。
3. **异构优势：** 在实际工程中，你可以让一个速度快、便宜的模型（如 GPT-4o-mini）负责**写代码（LLM 1）**，而让一个最顶级、逻辑最强、有视觉能力的大模型（如 GPT-4o 或 o1）负责**反思（LLM 2）**。这样既节省了成本，又保证了最终产出是“大师级”的。

## 八、如何评估反思模式

### 1. 客观评估

这类评估的结果是明确的、非黑即白的，不随人的意志为转移。

- **Code-based evals are easier (基于代码的评估更简单)：**原因： 代码是一个“非真即假”的世界。如果反思模式是为了修复 Bug，那么我们只需要运行代码。如果代码通过了编译器检查或单元测试，就说明反思成功了；反之则失败。**优势：** 自动化程度高，速度快，成本低。
- **Build a dataset of ground truth examples (构建基准真相数据集)：**做法： 开发者预先准备一套“考题”，并手动写下最完美、最准确的“标准答案”（Ground Truth）。**评估方式：** 将 AI 经过多轮反思后的最终产出与标准答案进行比对（例如计算相似度或关键信息覆盖率）。

### 2. 主观评估

这类评估针对的是没有标准答案的任务，比如写作文、写邮件或创意策划。

- **Use LLM as a judge (使用大模型作为裁判)：**做法： 这是目前 AI 评估中最流行的方法。请一个更强大的模型（如 GPT-4o 或 o1）来扮演“老师”的角色，给被测试模型的反思前后的表现打分。**为什么有效：** 强大的模型能够理解人类语言的微妙差异、逻辑的连贯性以及语气的专业度。
- **Rubric-based grading is better (基于评分量表的评分更好)：**核心建议： 不要只问裁判模型“好不好”，而要给它一份详细的**“质检清单（Rubric）”**。**例子：** “请从以下三个维度打分（1-5分）：1. 事实准确性；2. 语气的专业程度；3. 是否解决了用户的所有需求。”**价值：** 给定明确的标准后，AI 裁判给出的分数会更加一致、可重复且具有参考意义。

### 核心总结：

这页课件告诉我们，评估“反思”模式时要**“对症下药”**：

1. **能写代码测的，绝不写提示词测：** 优先选择客观评估，因为它最准确。
2. **必须主观评的，一定要给标准：** 如果必须用 AI 当裁判，一定要给它设计严密的评分细则（Rubric），防止 AI 裁判“随性打分”。

## 九、提示词工程的投资回报率 

它通过一个函数曲线图，直观地展示了开发者在优化 AI 表现时，投入的时间精力与最终获得的性能提升之间的关系。这张图传达了一个非常核心的观点：**单纯靠修改提示词（文字游戏）很快就会遇到瓶颈，真正的性能突破来自于工作流架构的改变（引入反思机制）。**

### 1. 🔴 红色曲线：无反思

- **模式：** 这是最基础的“直接生成（Zero-shot）”模式。
- **特征：** 曲线在初期上升很快，但很快就进入了**平台期（平缓期）**。
- **解读：** 当你开始写提示词时，稍微改进一下说法，性能会有明显提升。但很快你就会发现，无论你怎么字斟句酌、调整语气，模型的表现都很难再有实质性的进步。你已经触碰到了模型原生能力的“天花板”。

### 2. 🔵 蓝色曲线：带有反思

- **模式：** 这是智能体设计模式中的“自我反思”模式（模型生成后自查）。
- **特征：** 在红色曲线的分叉点处，通过引入反思逻辑，性能上限被显著拉高。
- **解读：** 与其花 10 个小时去抠提示词的形容词，不如花 1 个小时写一个“生成-反思-修改”的简单循环。这种**工作流（Workflow）的改变**能让同样的模型发挥出远超文字优化的水平。

### 3. 🟡 黄色曲线：带有外部反馈的反思

- **模式：** 这是最高阶的模式（例如模型写代码并在沙箱运行，根据报错修改）。
- **特征：** 性能上限最高，且持续增长的时间最长。
- **解读：** 这是“金标准”。当 AI 不仅仅是在脑子里想，而是能通过**调用工具**（如运行代码、搜索网页）获得真实世界的反馈时，它纠正错误的能力最强，产出的质量也最接近完美。

### 这页课件对开发者的核心价值 (ROI(Return on Investment)分析)

1. **避开“低效区”：** 很多开发者会陷入红色曲线的末端，花几天时间去试不同的提示词。这张图告诉你：**别试了，那是低 ROI 行为。**
2. **拥抱“高收益区”：** 当红色曲线变平时，你应该立刻转向**蓝色或黄色路径**。改变系统的架构（增加反思步骤或工具调用）才是真正能带来性能飞跃的高 ROI 投资。
3. **吴恩达的核心论点：** 正如他多次强调的，**“智能体工作流（Agentic Workflow）对性能的提升，往往比升级模型本身还要大。”**

## 十、辅助‘反思’模式的其他工具示例

它在智能体设计模式的背景下，展示了如何通过引入**外部客观反馈**来增强 AI 的自检能力。相比于让 AI 只在脑子里自我怀疑（自我反思），使用专门的工具能提供更硬核、更准确的纠错依据。

### 1. 案例一：规避提及竞争对手

- **挑战：** 企业级 AI 助手通常有安全策略，禁止在回答中直接提及或评价竞争对手（为了品牌安全或合规）。
- **例子：** AI 生成了“我们公司的鞋子比 RivalCo（竞争对手名）好”。
- **反馈来源：竞争对手名称的模式匹配（Pattern matching）**。**实现逻辑：** 系统后台运行一个简单的代码脚本，预设一个竞争对手名单。如果 AI 的初稿里命中了名单上的词，脚本会直接反馈给 AI：“你的回答中提到了受限的竞争对手名称，请修改后重新输出。”**优点：** 这种基于规则的硬性检查比让 AI “自觉遵守”要可靠得多。

### 2. 案例二：文章事实核查

- **挑战：** 大模型容易产生事实幻觉（编造日期、地点等）。
- **例子：** AI 写道“泰姬陵建于 1648 年”。
- **反馈来源：网络搜索结果（Web search results）**。**实现逻辑：** 智能体在写完初稿后，会提取其中的关键事实（如“泰姬陵 1648”），自动调用搜索引擎（如 Google/Bing API）进行比对。如果搜索到的准确时间是 1632-1653 年，系统会反馈给 AI：“根据搜索结果，泰姬陵的建造时间有误，请修正事实。”**优点：** 将回答锚定在外部真实世界的实时数据上，极大降低了幻觉率。

### 3. 案例三：控制输出长度

- **挑战：** 大模型对数字（如字数计数）极其不敏感，很难精准控制篇幅。
- **例子：** 用户要求写 50 字以内，但 AI 写了 100 字。
- **反馈来源：字数统计工具（Word count tool）**。**实现逻辑：** 系统调用一个简单的 Python 函数计算初稿字数。如果超限，反馈给 AI：“当前的字数是 100 字，超过了 50 字的限制。请在保留核心信息的前提下精简内容。”**优点：** 用精确的数学运算弥补模型对字数感知的弱点。

### 为什么需要外部工具辅助反思？

这页课件传达了智能体工程的一个进阶理念：**“以实证代替猜测”。**

1. **客观性：** AI 自身的反思有时会陷入逻辑循环或再次产生幻觉。外部工具（正则表达式、搜索 API、代码计算）提供的是**无可争辩的事实**。
2. **强制性：** 通过代码逻辑强制执行（如：字数不对就不通过），确保最终产出 100% 符合业务要求。
3. **闭环优化：** 这是上一页“黄色曲线（带外部反馈的反思）”的具体落地手段，也是 ROI（投资回报率）最高的优化方案。
