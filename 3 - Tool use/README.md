## 一、将代码作为工具

**所谓的“AI 工具”，本质上就是你写的一段普通的计算机代码。**

### 1. 核心定义

> **“工具仅仅是大模型（LLM）可以请求执行的代码。”**

- **打破神秘感：** 很多人以为给 AI 增加工具（比如搜索、算术、查数据库）需要非常复杂的 AI 技术。其实不然，AI 只是扮演了一个“发号施令”的角色，而真正的“活儿”还是由传统的编程代码完成的。
- **协作逻辑：**开发者写一段代码（函数）。开发者向 AI 描述这个函数的作用（通过注释或元数据）。AI 发现自己解决不了问题时（例如不知道现在的精确时间），它会输出一段指令说：“我想运行这个函数”。系统执行代码，并将结果喂回给 AI。

### 这页课件对开发者的实操启示

1. **AI 的“能力边界”由你定义：** 你想让 AI 具备什么能力（比如查公司的财务报表、控制家里的灯光、调用搜索引擎），你只需要写出对应的 API 调用代码或本地函数即可。
2. **注释即逻辑：** 在编写 Agent 工具时，**函数的描述（Description）比代码本身更重要**。如果你的描述写得不清楚，AI 就不知道在什么情况下该使用这个工具。
3. **安全提示：** 既然工具就是代码，那么在执行 AI 请求的代码时，必须注意**安全性**（比如在沙箱环境运行），防止 AI 被诱导执行危险的删除指令。

## 二、工具调用的实现原理

### 第一页：Defining tools syntax（定义工具语法）

这一页展示了开发者在前端编写代码的直观方式。

1. **定义函数（左侧）：**开发者写了一个普通的 Python 函数 get_current_time。**关键细节：** 注意其中的红色文字 """Returns the current time as a string"""。这被称为“文档字符串（Docstring）”。在 AI 开发中，这段话不再只是给人看的注释，而是**写给 AI 看的指令**。
2. **集成到模型（右侧）：**在调用大模型（如 GPT-4o）时，开发者只需在 tools 参数中直接放入函数名 [get_current_time]。
3. **核心结论：**底部的文字指出：函数被“自动描述”给了大模型。**直白点说：** AI 会阅读你的代码和注释，从而“理解”这个工具是干什么的，并自主决定在什么时候调用它。

### 第二页：Behind the scenes（幕后原理）

这一页揭示了上述代码在发送给大模型厂商（如 OpenAI）的服务器前，后台发生的**自动化转换过程**。

1. **映射关系：**红色框： 函数名 get_current_time 被自动提取，填入 JSON 数据的 "name" 字段。**绿色框：** 开发者写的注释（Docstring）被自动提取，填入 JSON 数据的 "description" 字段。
2. **JSON Schema（右上角）：**大模型底层并不直接读取 Python 代码。它读取的是一种标准格式，叫 **JSON Schema**。这份 Schema 告诉 AI：这里有一个叫 get_current_time 的函数。它的功能是“返回当前时间字符串”。参数（parameters）为空（因为这个函数不需要输入参数）。
3. **核心结论：**“名字和描述是自动添加的”。这意味着现代的 AI 开发框架（如课件中提到的 aisuite）极其智能。开发者只需要写好规范的 Python 代码和注释，框架就会自动完成从“可执行代码”到“AI 语义说明书”的转换。

### 总结：这给开发者的核心启示

这两页课件传达了智能体开发的一个**黄金准则**：

- **注释即逻辑：** 在编写 AI 工具时，你的**注释写得越清晰，AI 调用就越准确**。
- **如果 AI 总是选错工具：** 通常不是因为你的代码逻辑不对，而是因为你给函数的描述（Description）写得不够清楚，导致 AI 误解了工具的用途。

## 三、固定工具模式与动态写代码模式

### 第一张图：A simple calculator (固定工具模式)

在这种模式下，开发者为 AI 预设了一个有限的**工具箱**。

1. **场景：** 用户问“13.2 加 18.9”。
2. **工作流：**AI 识别出这是加法。AI 从预设的工具箱（加、减、乘、除）中选择 add 模块。计算结果 32.1 返回给 AI，AI 整合出最终回答。
3. **局限性：**当用户问“2 的平方根是多少？”时，工具箱里没有 sqrt（开方）工具。**痛点：** 难道要为每一种数学运算都手动写一个新工具吗？这太低效了。

### 第二张图：Alternative approach: Writing code (动态写代码模式)

这种模式是目前更高级、更通用的做法，通常被称为 **“代码解释器 (Code Interpreter)”** 模式。

1. **核心变化：** 不再给 AI 一堆细碎的函数（如加、减），而是给它一个**通用的编程环境**（Python）以及相应的指令。
2. **系统提示词 (System Prompt)：**明确要求 AI：“写代码来解决用户的问题。”要求将代码包裹在特定的标签中（如 <execute_python>）。
3. **工作流实测（以 2 的平方根为例）：**LLM 生成代码： AI 意识到需要开方，于是生成了一段代码：import math; print(math.sqrt(2))。**外部执行 (Execute Code)：** 系统捕获到这段代码，扔进 Python 沙箱里运行。**获取结果 (Result)：** 得到高精度的结果 1.41421356...。**最终生成：** AI 阅读计算结果，输出“2 的平方根约为 1.4142”。

### 深度总结与对比

这两页课件展示了 AI 智能体进化的两个阶段：

| 维度           | 固定工具模式 (第一图)                | 写代码模式 (第二图)                              |
| -------------- | ------------------------------------ | ------------------------------------------------ |
| **灵活性**     | **低**。只能处理预设好的加减乘除。   | **极高**。可以处理任何能用 Python 表达的问题。   |
| **开发工作量** | **高**。需要为每个功能写接口和文档。 | **低**。只需给 AI 一个 Python 环境和一段提示词。 |
| **安全性**     | 较高（工具是受控的）。               | 较低（需要严格的沙箱环境防止 AI 运行恶意代码）。 |
| **能力边界**   | 受限于工具箱的大小。                 | 理论上无限大（数据处理、绘图、科学计算）。       |

### 总结

这页课件告诉我们：**“写代码”是智能体最强大的元能力。** 相比于给 AI 准备成百上千个死板的“工具”，给它一个“代码执行环境”，让它根据问题**即时编写逻辑**，才是构建全能智能体的终极路径。

## 四、安全的代码执行

### 1. 核心风险：不要在“裸机”上运行 AI 的代码

> *“在沙箱之外运行可能是危险的”*

- **危险案例（图中黑底文字）：**
  图中展示了一个 AI 的反思摘要：“你是对的——那是一个极其愚蠢的错误。我永远不应该在项目目录里使用 rm *.py 指令。”**解读：** rm *.py 是 Linux 系统中删除所有 Python 文件的命令。如果 AI 产生幻觉，或者为了完成某个任务（比如“清理旧代码”）而写了这一行代码，并且你的程序**直接在服务器的主系统里运行了它**，那么你的所有源代码都会被 AI 一瞬间删光。**更多风险：** AI 可能会写出上传系统私钥的代码、启动耗尽内存的死循环、或者尝试攻击内网其他服务器。

### 2. 解决方案：使用沙箱 (Sandboxes)

> *“沙箱可以帮助防止灾难性的错误”*

- **什么是沙箱：** 简单来说，就是一个**逻辑隔离的“塑料房”**。AI 在里面可以随意写代码、删文件、装软件，但它无法触碰到外面的真实系统。一旦任务结束，这个塑料房会被直接销毁重置。
- **主流技术工具：**Docker:**业界最流行的容器化技术。**作用： 为 AI 的代码运行提供一个独立的镜像环境。即使 AI 运行了 rm -rf /，也只能毁掉这个临时的容器，对宿主机毫发无伤。**E2B (右上角 Logo):**这是一个专门为 AI 智能体设计的**云端沙箱环境**。**作用：** 它提供了一套非常方便的 API，让开发者能一键给自己的 AI 智能体开辟一个带有网络连接、支持各种编程语言的安全执行空间。

### 实操总结：给 AI 开发者的一条“军规”

这页课件告诉我们：**永远不要相信大模型生成的代码是安全的。**

1. **物理隔离：** 所有的代码执行环节（Code Interpreter 模式）必须在沙箱或容器中进行。
2. **权限最小化：** 沙箱不应该有权访问你的数据库密钥、环境变量或局域网敏感资产。
3. **超时与资源限制：** 必须设定代码运行的最大时长和内存上限，防止 AI 写出“死循环”导致服务器账单爆炸。

## 五、模型上下文协议

### 1. 核心矛盾：碎片化的连接 (Spaghetti Connections)

图中左侧展示了 **Apps（AI 应用，如 Claude、ChatGPT 等）** 和右侧的 **Tools（办公工具/数据库，如 Slack、Google Drive、GitHub 等）**。

- **传统模式的问题：** 每一家 AI 应用如果想接入这些工具，都需要针对每一个工具单独开发一套代码。
- **数学代价 (**`**`m×n`**`**)：** 假设你有 `m` 个应用和 `n` 个工具，整个行业就需要维护 `m` 乘以 `n` 条连接路径。
- **后果：** 这种“蜘蛛网”式的连接导致开发效率极低，且工具无法在不同平台间通用。

### 2. MCP 的革命性解决方案：标准化 (The Standard)

MCP 协议本质上是在应用和工具之间建立了一层**通用的“翻译官”**。

- **MCP 服务器 (MCP Server)：** 工具开发者（如 Slack）只需要按照 MCP 标准写一次接口。
- **MCP 客户端 (MCP Client)：** 应用开发者只需要让自己的应用支持 MCP 协议。
- **数学飞跃 (**`**`m+n`**`**)：** 复杂度从乘法变成了加法。
  - 这意味着：只要一个工具做好了 MCP 适配，**所有的** AI 应用都能立刻使用它；反之，一个应用支持了 MCP，它就能瞬间连接**所有的** MCP 工具。

### 3. 为什么 MCP 对智能体如此重要？

这页课件传达了 AI 基础设施进化的一个重大信号：

1. **打破壁垒（Interoperability）：** 以前你的 AI 助手可能只能在特定平台用特定功能。有了 MCP，你的个人 AI 可以无缝地跨越 Slack 查消息、去 GitHub 提代码、去 PostgreSQL 查报表。
2. **降低门槛：** 开发者不再需要成为“接口集成专家”，只需要专注于 AI 的逻辑设计。
3. **智能体的“通用插座”：** MCP 就像是 AI 领域的 **USB 接口标准**。它让 AI 真正具备了大规模连接现实世界数据的能力，而不再被困在“聊天框”里。

## 六、使用预构建的客户端和服务器

### 1. 客户端 —— “能用 AI 的软件”

MCP 客户端是那些**“发起请求”**的 AI 应用。课件展示了目前支持 MCP 协议的主流软件：

- **Cursor:** 目前全球最火的 AI 编程编辑器，它可以通过 MCP 直接访问你的本地文件系统或数据库。
- **Claude Desktop:** Anthropic 官方的桌面版 Claude，可以通过 MCP 连接外部工具（如 Slack、GitHub）。
- **Windsurf:** 另一款新兴的 AI 编码助手，同样支持 MCP 生态。
- **Your App:** 意味着你也可以把自己的应用变成一个 MCP 客户端。

### 2. 服务器 —— “被 AI 使用的工具”

MCP 服务器是那些**“提供数据和功能”**的后端接口。课件列举了目前已有的现成 MCP 服务器：

- **Slack:** 允许 AI 读取或发送消息。
- **Google Drive:** 允许 AI 检索你的云端文档。
- **GitHub:** 允许 AI 查看代码库、提交 Issue 或 PR。
- **PostgreSQL:** 允许 AI 直接对数据库执行查询。
- **Your Server:** 鼓励开发者为自己的私有工具开发 MCP 服务器，从而瞬间接入上述所有的 AI 客户端。

### 为什么这页内容对开发者极具吸引力？

1. **“即插即用”的生态：**
   以前，如果你想让 Claude 帮你总结 Slack 里的讨论并存入 GitHub，你需要写大量的集成代码。现在，你只需要下载现成的 **MCP Slack Server** 和 **MCP GitHub Server**，然后在 **Claude Desktop** 里配置一下，这件事立刻就办成了。
2. **打破孤岛：**
   以前 AI 客户端（如 Cursor）只能看代码。现在通过 MCP，你可以让 Cursor 实时查询数据库（PostgreSQL）的结构，从而写出更准确的 SQL 代码。
3. **未来的开发标准：**
   如果你开发了一个新工具（Your Server），只要你支持 MCP 协议，全球所有的顶级 AI 编程器（Cursor, Windsurf 等）都能立刻学会如何使用你的工具，无需你逐一去谈合作。
