## 一、将行动计划格式化为 JSON

### 1. 系统提示词：规则设定

左侧黑框是给 AI 设定的“行为准则”。它不再让 AI 随便聊天，而是下达了严密的指令：

- **工具授权：** 告知 AI 可以使用哪些工具（{description of tools}）。
- **格式强制：** 要求必须创建一个 **JSON 格式** 的逐步计划（step-by-step plan）。
- **结构定义：** 规定了 JSON 里的每一个步骤必须包含：步骤编号（step number）、描述（description）、工具名称（tool name）和参数（args）。

### 2. 用户提问：模糊的需求

红色框代表用户的原始需求：

> “你们有 100 美元以下、有库存的圆框太阳镜吗？”

**分析：** 这是一个复杂需求，包含了多个过滤条件（圆框、有库存、价格低于100）。模型无法通过一次简单的搜索完成。

### 3. LLM 的角色：规划师

大模型（LLM）接收到上述指令后，不再直接回答“有”或“没有”，而是像一个**项目经理**一样开始拆解任务。

### 4. 输出结果：结构化的行动方案

右侧灰色部分是 AI 吐出的 JSON 数据。这才是这页课件的核心：

- **步骤 1 (Step 1)：**描述： 寻找圆框太阳镜。**工具：** 调用 get_item_descriptions。**参数：** 搜索关键词 "round sunglasses"。
- **步骤 2 (Step 2)：**描述： 检查可用库存。**工具：** 调用 check_inventory。**参数：** 输入来自第一步的结果（"results from step 1"）。这里展示了**逻辑依赖关系**。

### 为什么这页课件对开发者非常重要？

1. **机器可读性：** 自然语言（文字）对程序来说很难处理。但 JSON 是一种标准格式，你的程序代码可以直接读取 tool 和 args 字段，然后自动化地去调用对应的数据库接口。
2. **可靠性：** 强制要求 AI 先写出计划，能让它在执行前理清思路，极大减少了处理复杂逻辑时的幻觉（Hallucination）。
3. **可调试性：** 如果 AI 找错了东西，开发者可以通过查看这个 JSON 计划，快速判断出是“第一步搜错了”还是“第二步查错了”。

## 二、使用工具进行规划的挑战

### 1. 任务背景

- **用户问题（红色框）：** “哪个月份的热巧克力销售额最高？”
- **目标数据：** 一个名为 coffee_sales.csv 的表格，包含日期、价格、咖啡名称和规格。
- **可用工具（Tools）：** 开发者提供了一组非常基础的数据处理工具，如：filter_rows（过滤行）、get_column_mean（求列平均值）、sum_rows（求行总数）等。

### 2. 暴露出的“挑战”：笨拙的规划逻辑

观察右侧 AI 生成的 **6 步计划**，你会发现这是一种非常低效的“笨办法”：

- **步骤 1 & 2：** 过滤出 1 月份的热巧克力数据 `→→` 计算平均值。
- **步骤 3 & 4：** 过滤出 2 月份的热巧克力数据 `→→` 计算平均值。
- **步骤 5：** **重复 12 次！** 为 3 月到 12 月重复上述过程。
- **步骤 6：** 手动比较这 12 个结果，找出最高的一个。

### 深度分析：为什么这被称为一个“挑战”？

这页课件想要传达三个核心工程痛点：

1. **路径爆炸（Complexity Explosion）：**
   对于人类或成熟的程序来说，这只需要一行代码（比如 Python 的 groupby）。但因为工具给得太细、太死板，AI 被迫写出了一个长达 20 多步的“流水账”计划。
2. **高延迟与高成本：**
   每一步工具调用都意味着一次往返通信。要回答这个问题，AI 需要和服务器来回交互 20 多次。这会让用户等很久，且消耗大量 Token。
3. **脆弱性（Fragility）：**
   步骤越多，出错的概率就越高。如果 8 月份的数据检索失败了，整个计划可能就崩盘了。
4. **缺乏“抽象思维”：**
   这展示了 LLM 在使用原子化工具时的局限性——它像一个只会搬砖的工人，而不知道如何开吊车。

### 实战建议：如何解决这个问题？

这页课件实际上是为后面要讲的**“写代码（Writing Code）”**模式做铺垫：

- **更好的方案：** 不要给 AI 提供 filter_rows 这种细碎的工具。
- **终极方案：** 给 AI 一个 **Python 执行环境**（Code Interpreter）。如果是写代码模式，AI 只需要 1 个步骤：写一段 Python 脚本，用 pandas 库直接算出结果并返回。**1 步 vs 24 步**，这就是效率的质变。

## 三、预设固定工具集

### 1. 现状：为每个需求增加特定工具

图中展示了当用户提出新类型的需求时，开发者不得不陷入“头痛医头、脚痛医脚”的循环：

- **新需求 1：** “上周有多少次唯一的交易？” `→→` **必须新增工具：** get_unique_entries（获取唯一条目）。
- **新需求 2：** “最后 5 笔交易的金额是多少？” `→→` **必须新增工具：** get_last_N_values（获取最后 N 个值）。
- **趋势：** 随着业务变复杂，你的工具箱（Tools box）会变得越来越臃肿，开发者需要不断地编写、维护和向 AI 描述这些新工具。

### 2. 核心痛点 (右侧要点)

课件总结了这种“原子化工具”模式的三大致命伤：

- **Brittle (脆弱性)：**系统非常“脆”。如果用户换一种问法，或者数据格式稍微变一点（比如日期格式从 YYYY-MM-DD 变成 DD/MM/YY），这些写死的工具可能就无法工作了。AI 在组合这些细碎工具时，只要其中一步出错，整个链路就会崩盘。
- **Inefficient (低效)：**正如上一页课件展示的，为了算一个“月度最高销售”，AI 可能会调用几十次小工具。**代价：** 大量的 API 往返请求会导致系统反应极慢，且产生高昂的 Token 费用。
- **Continuously dealing with edge cases (无休止地处理边缘情况)：**开发者会发现自己陷入了无穷无尽的补丁工作中。今天为了处理“空值”写个逻辑，明天为了处理“重复项”加个参数。你永远无法预穷举用户所有的提问方式。

### 💡 深度总结：这页课件想表达什么？

这页课件实际上是在推翻 **“为 AI 准备万能工具箱”** 的陈旧思维。

**它在引导你思考：**
与其给 AI 准备 100 个专门处理表格的小工具（过滤、求和、去重、排序...），为什么不直接给它一个**通用的编程环境（Python / Code Interpreter）**呢？

- **对比：**固定工具： AI 像个只会按按钮的猴子，没按钮就干不了活。**写代码模式：** AI 像个真正的程序员。无论需求怎么变，它只要写一段逻辑严密的 Python 代码就能解决。

### 总结

这页课件是**“从工具调用（Tool Use）进化到代码执行（Code Execution）”**的论据。它告诉我们：在处理复杂、多变的数据任务时，**灵活性（代码）远比数量（固定工具）重要**。

## 四、使用代码进行规划能显著提升智能体的性能

### 1. 三种行动模式的定义

图表右侧列出了三种 AI 智能体执行任务的方式：

- **💜 Text as Action (文本作为行动)：** AI 用纯自然语言描述它要做什么。这是最原始、最模糊的方式，性能表现也最差（图中紫色条最短）。
- **🩷 JSON as Action (JSON 作为行动)：** AI 输出结构化的 JSON 格式指令（即我们常说的“函数调用” Function Calling）。这比纯文本更严谨，性能有明显提升（图中粉色条居中）。
- **🩵 Code as Action (代码作为行动)：** AI 直接编写并执行 Python 代码来完成任务。**这是表现最好的模式**（图中青色条最长）。

### 2. 核心实验数据发现

观察图表中的数据，我们可以得出几个震撼的结论：

- **全线胜出：** 无论使用的是哪个模型（从最强的 GPT-4 到开源的 Llama-2），**青色条（代码模式）永远是三者中最长的。**
- **质的飞跃：**以 **gpt-4-1106-preview** 为例，使用 JSON 模式的成功率约在 50% 多，但一旦切换到**写代码模式**，成功率直接飙升到了接近 **75%**。即使是较弱的模型（如 **gpt-3.5**），其代码模式的表现甚至能追平或超过更强模型在文本模式下的表现。
- **逻辑优势：** 研究表明，将“规划”过程转化为“编写代码”的过程，能够激发大模型更强的逻辑推理能力。

### 为什么“写代码”比“发指令(JSON)”更好？

这页课件回答了前面几页提出的挑战。代码模式优于 JSON/工具调用模式的原因包括：

1. **处理复杂逻辑的能力：** JSON 只能表达“调用 A，参数是 B”。但代码可以表达**循环（Loops）**、**条件判断（If-Else）和变量存储**。
2. **极高的灵活性：** 就像之前咖啡销售的例子，代码可以用一行 df.groupby().max() 解决问题，而不需要调用 24 次原子工具。
3. **模型更“擅长”代码：** 顶级大模型（如 GPT-4）在训练数据中包含了海量高质量的代码，它们对代码逻辑的理解往往比对零散工具说明书的理解更深刻、更严谨。

## 五、多智能体规划

智能体设计模式中的**“多智能体规划（Planning with Multiple Agents）”**。它通过一个实际的业务场景，演示了如何将一个复杂的任务拆解并分派给一个由多个专门 AI 组成的“虚拟团队”来协作完成。

### 核心知识点与启示

1. **角色分化（Specialization）：**
   通过给不同的 Agent 设定不同的 Persona（人格/角色），可以强制模型从特定的专业视角思考。研究员会关注数据，设计师会关注色彩和构图。
2. **任务拆解（Decomposition）：**
   将大任务拆成独立、有序的小任务。每一步的输入都依赖上一步的输出，这种逻辑链条让 AI 生成的内容极具严谨性。
3. **人机协作模式：**
   这种架构通常被称为 **“经理-员工”架构**。主模型负责宏观规划和最终质检，子模型负责具体执行。
4. **容错性：**
   如果“研究员”找的信息不对，经理可以在第一步就打回重做，而不会等到最后写完报告才发现基础数据错了。

## 六、线性计划

### 1. 核心结构：信息流水线

图中有三个核心角色（Agent），它们通过箭头连接，形成了一个单向流动的价值链：

- **Researcher（研究员）** `**`→`**` **Graphic Designer（设计师）：**
  - **动作：** 研究员首先完成市场调研（比如当前流行什么颜色的镜片、目标受众是谁）。
  - **传递：** 研究报告被发给设计师。没有这些背景数据，设计师可能会闭门造车，画出不符合市场的图。
- **Graphic Designer（设计师）** `**`→`**` **Writer（撰稿人）：**
  - **动作：** 设计师根据调研结果创造视觉元素或广告图。
  - **传递：** 设计方案和调研报告一起发给撰稿人。

### 2. 关键路径：跨层信息共享

注意图中底部那根从 **Researcher** 直接指向 **Writer** 的长箭头。这在智能体架构中至关重要：

- **逻辑：** 撰稿人在写最后的“营销策划案”时，不仅需要看设计师做出的精美图表，还需要**直接参考**研究员最初搜集到的第一手市场数据（如：竞争对手定价、消费者反馈）。
- **目的：** 确保最终产出（Final report）不会因为中间环节的传递而导致信息衰减或变形。

### 💡 核心知识点：线性计划的特点

这页课件定义的 **"Linear plan"（线性计划）** 具有以下几个重要属性：

1. **顺序执行：** 必须先有 A，才有 B。这种结构逻辑清晰，非常适合那些有标准作业程序（SOP）的任务。
2. **累积上下文：** 每往后走一步，Agent 接收到的信息量都在增加。Writer 手里的资料是最全的，因为它承接了 Researcher 和 Graphic Designer 的双重产出。
3. **确定性高：** 流程是预先设计好的，不容易跑偏。

## 七、中心化编排模式/星型通信模式

### 1. 核心结构：枢纽与分支

图中展示了一个以“市场经理”为核心的辐射状结构：

- **中心节点：** Marketing Manager（编排者/主控模型）。
- **分支节点：** Researcher（研究员）、Graphic Designer（设计师）、Writer（撰稿人）。
- **通信方式：** 注意箭头是**双向的**，且所有箭头都指向中心。这意味着子智能体之间**不直接对话**，所有的信息交换、任务分派和结果汇总都必须经过“经理”。

### 2. 工作逻辑：指挥与反馈

在这个模式下，任务是这样完成的：

1. **分派任务：** 经理先给“研究员”下令，让他去搜集资料。
2. **结果回收：** 研究员做完后，把报告交还给“经理”。
3. **二次分派：** 经理检查研究报告，如果没问题，再把这份报告发给“设计师”，并告诉他根据报告画图。
4. **最后整合：** 设计师把图给“经理”，撰稿人把文案给“经理”，由“经理”进行最后的拼装和审核。

### 这种模式的优势（为什么要这么设计？）

相比于“线性模式”，这种中心化架构有三个巨大的工程优势：

1. **更强的质量控制（Quality Gate）：**
   经理可以在每一步执行完后进行**反思（Reflection）**。如果研究员找的资料太烂，经理可以直接命令他“重做”，而不会让错误的信息流向下游的设计师。
2. **动态任务调度：**
   经理可以根据前一步的结果灵活决定下一步。例如，如果研究员发现市场上根本不需要太阳镜，经理可以立刻终止流程，或者改让设计师去画遮阳伞。
3. **防止“上下文污染”：**
   每个子 Agent 只接收到经理发给它的、与其任务最相关的精简信息，而不需要像线性模式那样背负着前面所有人产生的冗长历史记录，从而提高了小模型的准确度。

## 八、深度层级模式与全连接模式

### 1. 左侧：深度层级模式

这种模式模拟了现代大型企业的**组织架构（Organization Chart）**。

- **结构特点：**多层级管理： 存在“经理、主管、员工”三层结构。**局部自治：** 底层的员工（如 web researcher）只向他的直属上司（researcher）汇报。
- **具体分工案例：**Researcher 组： 组长负责汇总，手下有两个兵，一个专门负责搜网页，一个专门负责做事实核查。**Writer 组：** 组长负责统稿，手下有美工（style writer）负责润色，引用专家（citation checker）负责查来源。
- **优势：**极高的专业度： 每一个子节点都是极度聚焦的专家模型。**降低复杂度：** 顶层的市场经理不需要处理搜索细节，他只看汇总后的研究报告。这保护了顶层模型的上下文空间（Context Window）。

### 2. 右侧：全连接模式

这种模式模拟了敏捷团队的**“圆桌会议”**。

- **结构特点：**去中心化： 每一个智能体都可以直接和其他任何人对话。**信息共享：** 所有人都在同一个频道里。
- **具体协作逻辑：**设计师可以直接问研究员：“你那个数据来源可靠吗？”撰稿人可以直接告诉经理：“设计师的图和我的文案不搭，我们需要重开。”
- **优势：**沟通效率极高： 减少了中间环节的传话。**动态协作：** 非常适合需要高度创意、反复碰撞火花的任务。

### 核心总结与决策建议

这两张图揭示了构建智能体系统的工程美学：

1. **规模决定架构：**如果任务非常巨大（如写一本书、做一个完整的 App），选 **Deeper hierarchy**。通过层层嵌套来分担逻辑压力。如果任务需要灵感碰撞或多方对齐，选 **All-to-all**。
2. **权衡（Trade-off）：**层级模式**更稳健、更好管理（Manageable），但可能会因为信息传递太久而产生延迟。**全连接模式更灵活，但非常容易导致“吵群”现象（信息冗余），甚至让模型陷入死循环。

## 九、核心总结

### 1. 为什么要使用智能体 AI

- **核心逻辑：** 传统的“直接生成（Zero-shot）”模式很快会遇到性能天花板。
- **结论：** 智能体化通过将任务拆解为**迭代的工作流**，让现有的模型（哪怕是较弱的模型）通过“勤能补拙”的方式，在复杂任务上的表现反超更强大的模型。它标志着 AI 从“只会聊天”向“能够办事”的质变。

### 2. 反思设计模式 (Reflection design pattern)

- **核心逻辑：** 模拟人类“写稿-校对-修改”的过程。
- **关键点：** 这是提升 AI 准确率最立竿见影的方法。无论是**自我反思**还是引入专门的**评审模型（Critic）**，通过多轮迭代和针对性的“挑刺”，可以极大地减少事实错误和逻辑漏洞。

### 3. 工具调用 / 函数调用 (Tool use / function calling)

- **核心逻辑：** 给 AI 穿上“手和脚”。
- **关键点：** 通过 **JSON Schema** 协议，将自然语言指令转化为计算机代码。这让 AI 具备了获取**实时信息**（搜索）、处理**精密计算**（计算器/代码运行）以及**操作系统**（读写数据库、发邮件）的能力，打破了模型自身的局限。

### 4. 评估与错误分析 (Evals, error analysis)

- **核心逻辑：** AI 开发不能靠“感觉”，要靠“实验数据”。
- **关键点：**评估： 结合客观的代码测试（Code-based）和主观的 AI 裁判（LLM-as-judge）。**分析：** 培养查看**链路追踪（Traces）**的习惯。当系统失败时，通过分析 Trace 确定到底是“检索不到位”还是“模型想偏了”，从而精准投入研发精力。

### 5. 规划与多智能体系统 (Planning, multi-agent systems)

- **核心逻辑：** 应对极度复杂的业务逻辑。
- **规划：** 让 AI 扮演“项目经理”，将模糊需求转化为结构化的 JSON 步骤或可执行的 Python 代码。
- **多智能体：** 模拟“人类专家团队”。通过**线性、星型、层级或全连接**等通讯模式，让专门的 AI 角色（如程序员、测试员、分析师）分工协作，处理单个模型无法处理的大型工程。

### 结语：这模块的实战价值

这页课件勾勒出了 **“数字员工”** 的完整画像：

1. 它有**办事流程**（Agentic Workflows）；
2. 它懂得**自我质检**（Reflection）；
3. 它会**使用工具**（Tool use）；
4. 它能**服从管理和持续进步**（Evals）；
5. 它会**做计划并能带团队**（Planning & Multi-agent）。
